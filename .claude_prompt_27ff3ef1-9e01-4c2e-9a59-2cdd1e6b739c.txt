## Original Context
**Title**: Phase 2: Thread view for messages domain
**Description**: Build the page-level thread view component that assembles message chunks from a single source into a conversation-style presentation, with reply hierarchy derived from domain metadata.

## Requirements
- Display the message thread root chunk at the top, identified by `domain_metadata.is_thread_root === true`
- Display reply messages below the root, ordered by `domain_metadata.timestamp` ascending
- Build a reply tree using `domain_metadata.in_reply_to` → `domain_metadata.message_id` relationships and visually indent nested replies
- Support arbitrarily nested reply chains (replies to replies)
- Degrade gracefully when reply chunks reference missing message IDs (render as flat list under root, no error state)
- Display per-message: sender, recipients, subject line, timestamp, and rendered body content (not raw JSON)
- Visually distinguish the thread root from reply messages
- Encode `thread_id` in the URL so the view is directly linkable
- Display a 'View Raw Chunks' link back to the browser chunk table for this source

## Design Guidance
**Component location:** `ui/src/views/ThreadView.tsx`

**Props interface (from registry):**
```typescript
interface DomainViewProps {
  sourceId: string;
  chunks: ChunkResponse[];   // all chunks for the source, already fetched
  source: SourceDetailResponse;
}
```

**MessageMetadata fields (from backend `storage/models.py`):**
```typescript
// Accessed via chunk.domain_metadata (typed as Record<string, unknown> | null)
interface MessageMetadata {
  thread_id: string;
  message_id: string;
  sender: string;
  recipients: string[];     // serialized from tuple
  timestamp: string;        // ISO 8601
  in_reply_to: string | null;  // references another message_id
  subject: string | null;
  is_thread_root: boolean;
}
```

**Assembly algorithm:**
1. Cast `chunk.domain_metadata` to `MessageMetadata` with null checks
2. Find root: first chunk where `is_thread_root === true`; if none, use chunk with lowest `chunk_index`
3. Build a `Map<message_id, chunk>` for O(1) lookup
4. Build reply tree: for each chunk, if `in_reply_to` is set and exists in the map, attach as child; otherwise treat as top-level
5. Render root, then recursively render children with CSS `paddingLeft` incremented per depth level (e.g., `depth * 24px`)
6. Sort siblings by `timestamp` ascending

**Visual structure:**
```
[Thread root - full width card]
  Subject: Re: Meeting Notes
  From: alice@example.com  →  bob@example.com
  2024-01-15 14:32
  ─────────────────
  [message body rendered via <MarkdownContent>]

  [Reply - indented 24px]
    From: bob@example.com
    2024-01-15 15:01
    ─────────────────
    [reply body]

    [Reply to reply - indented 48px]
      From: carol@example.com
      ...
```

**Shared primitives to use:**
- `<Timestamp value={meta.timestamp} granularity='datetime' />`
- `<MarkdownContent content={chunk.content} />` for message body
- `<ChunkBoundary />` between top-level reply cards

**Data source:** `useSourceChunks(sourceId)` hook — each email thread is a single source in the data model. No new API endpoints required.

**URL state:** The `thread_id` comes from the `$sourceId` route parameter (source = thread). No additional URL search params needed for the basic view.

### Issues Found

#### High Priority (Should Fix)

- **Root fallback does not actually select lowest `chunk_index`**: The comment on line 75 says "Find root: is_thread_root or lowest chunk_index," and the requirement states the same. However, the fallback just takes `chunksByMetadata[0]` — the first element after `.map().filter()`. Array ordering after `.map()` preserves the order of the input `chunks` array, which is not guaranteed to be sorted by `chunk_index`. If no chunk has `is_thread_root === true`, the code should explicitly find the chunk with the lowest `chunk_index` rather than assuming array order. The fix is straightforward: when no `is_thread_root` is found, iterate to find `min(chunk_index)`.

- **Dead `ThreadView` export remains in `placeholders.tsx`**: `placeholders.tsx:9` still exports `export const ThreadView`. The registry no longer imports it from there (it now imports from `./ThreadView`), so this is dead code that will confuse future maintainers and could cause accidental import of the wrong component. It should be removed as part of this PR since the registry change is in scope.

- **`extractMessageMetadata` try/catch provides false safety**: The function wraps property accesses in a `try/catch`, but none of the operations inside (`as` casts, `??`, `Boolean()`) can throw. TypeScript `as` casts are compile-time only and never throw at runtime. If `meta.message_id` is `undefined`, it will silently produce `undefined as string` — not throw — and the returned object will have `message_id: undefined`, which will cause silent downstream failures (e.g., `nodeMap.set(undefined, ...)`, broken key lookups). The function should validate that required fields (`message_id`, `sender`, `timestamp`) are actually present and are strings, returning `null` if not. Remove the misleading try/catch in favor of explicit checks.

#### Advisory (Out of Scope / FYI)

- **`key` prop on the root `<div>` in `MessageCard` is unnecessary**: At `ThreadView.tsx:155`, `<div key={metadata.message_id} ...>` sets a `key` on the outermost element of `MessageCard`. This `key` has no effect here because `key` is meaningful only when rendering lists of sibling elements via `.map()`. The actual list key is already correctly set on the wrapper `<div key={child.metadata.message_id}>` at line 205. Not harmful, but slightly misleading.

- **No depth limit for deeply nested reply chains**: The recursive rendering supports "arbitrarily nested reply chains" as required, but extremely deep chains could cause layout issues (content pushed off-screen by `depth * 24px` padding). This is an edge case unlikely to occur with real email threads, so it's fine for now. Consider capping indentation depth in a future iteration if needed.

- **`messageMap` is redundant with `nodeMap`**: `messageMap` (line 85) maps `message_id → [chunk, meta]` and `nodeMap` (line 90) maps `message_id → MessageNode`. The only use of `messageMap` after construction is `messageMap.has(meta.in_reply_to)` on line 111, which could equivalently use `nodeMap.has(meta.in_reply_to)`. This would eliminate one data structure. Minor cleanup, not blocking.

- **`useNavigate` called unconditionally when only used in one branch**: `useNavigate()` is called at the component top level even when `rootNode` is null (the fallback branch). This is fine per React rules of hooks and is not a bug, just a minor observation.

### Summary

The implementation is well-structured overall. The tree-building algorithm is clear, shared primitives are used correctly, the component integrates properly into the domain view registry, and the visual structure matches the design spec. TypeScript types check cleanly.

Three issues need attention before merge: (1) the root fallback should explicitly find the chunk with the lowest `chunk_index` rather than relying on array order, matching both the spec and the code's own comment; (2) the dead `ThreadView` placeholder export should be removed; and (3) the `extractMessageMetadata` function should validate required fields rather than relying on a try/catch that can never trigger.
