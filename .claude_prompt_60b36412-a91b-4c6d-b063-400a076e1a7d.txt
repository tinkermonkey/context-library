- Enhanced `register_source()` method to detect domain changes on re-registration
- When a source is re-registered with a different domain:
  1. Check the current domain in the `sources` table (lines 495-500)
  2. If the domain differs from the new domain, update all chunks for that source (lines 513-520)
  3. Then update the source configuration as normal
- Updated method docstring to document this new behavior (lines 480-481)

### **Key Design Decisions**

- **Minimal, focused changes**: Only modified what's necessary to fix the issue
- **Automatic migration**: Domain changes are handled transparently during re-registration, requiring no manual intervention
- **Backward compatible**: Existing databases work correctly without requiring special migration scripts

#### High Priority (Should Fix)

- **Vector store domain metadata not updated on domain change**: The new chunk migration logic in `document_store.py:513-520` correctly updates the `domain` column in the SQLite `chunks` table when a source is re-registered with a different domain. However, the vector store (ChromaDB/LanceDB) also stores `domain` as metadata on each vector (see `chromadb_store.py:68-73` and `lancedb_store.py:24-32`), and both use it for domain-based filtering in `search()`. The `register_source` method only touches SQLite — it does not update vector store metadata or write sync log entries that would trigger a reconciliation. This means after a domain migration:
  - SQLite chunks: `domain = 'documents'` (correct)
  - Vector store metadata: `domain = 'notes'` (stale)
  - A query filtering by `domain_filter=Domain.DOCUMENTS` will miss these chunks in the vector store
  - A query filtering by `domain_filter=Domain.NOTES` will return stale matches
  
  **Recommendation**: The simplest correct approach is to record delete + insert sync log entries for all affected chunk hashes when the domain changes. This way, the existing `get_chunks_pending_sync()` / `get_chunks_pending_deletion()` reconciliation path will rebuild the vectors with the correct domain. Alternatively, since this is a one-time migration event (NOTES→DOCUMENTS), you could document that a full vector store rebuild is required after the upgrade and skip runtime sync — but that shifts the burden to operators.

- **Missing test for chunk domain migration**: The existing test `test_register_source_updates_domain_and_adapter_id_on_reregister` (`test_document_store.py:2268-2343`) only verifies that the `sources` table is updated. It does not create any chunks before re-registration and therefore does not verify that the new `UPDATE chunks SET domain = ?` statement actually works. A test should:
  1. Register a source with `Domain.NOTES`
  2. Create a source version and insert chunks with `domain='notes'`
  3. Re-register the source with `Domain.DOCUMENTS`
  4. Assert all chunks now have `domain='documents'`
  
  Without this test, the chunk migration path has zero coverage and regressions would go undetected.

#### Advisory (Out of Scope / FYI)

- **`register_source` does not have access to vector store**: This is an architectural observation. `DocumentStore` (SQLite layer) has no reference to the vector store, so it fundamentally cannot update vector metadata. The coordination would need to happen at a higher level (e.g., pipeline or a new migration utility). This is a design constraint, not a bug in this PR per se, but it explains why the vector store inconsistency exists.

- **Existing chunks will use Documents domain chunking on next ingestion**: On the next pipeline run, the differ will re-chunk filesystem content using the `DocumentsDomain` chunker instead of `NotesDomain`. Since chunk hashes are content-based but context headers differ by domain, this will naturally produce new hashes and the old notes-domain chunks will be retired. This means the SQLite chunk domain update is somewhat transient — it corrects metadata for the interim period before re-ingestion. The vector store inconsistency is the more pressing gap since stale domain values affect search filtering until vectors are rebuilt.

### Summary

The PR correctly addresses both stated requirements: `FilesystemHelperAdapter` now uses `Domain.DOCUMENTS` (consistent with `FilesystemAdapter`), and `register_source` migrates chunk domains in SQLite on re-registration. The code is clean and the docstring update is appropriate.

However, the vector store consistency gap is a real functional issue — domain-filtered searches will return incorrect results for migrated chunks until the vector store is rebuilt. This needs to be addressed, either by adding sync log entries for affected chunks or by documenting that a vector store rebuild is required. Additionally, the chunk migration path lacks test coverage and should have a dedicated test.
