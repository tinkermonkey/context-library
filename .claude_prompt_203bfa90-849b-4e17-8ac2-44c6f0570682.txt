## System Architecture

### Problem Abstraction

Phase 3 introduces two new semantic domains — **Tasks** and **Events** — each with distinct structural characteristics that differentiate them from the existing Notes and Messages domains. Tasks are stateful, potentially hierarchical entities with lifecycle tracking (status, priority, dependencies). Events are temporal entities with scheduling semantics (start/end times, duration, recurrence, participants). Both domains require new metadata models, domain chunkers, and adapters that span local macOS clients and standards-based protocols.

The core architectural question is: how do these new domains and their adapters integrate with the existing pipeline without introducing unnecessary abstractions or violating the established adapter-domain contract?

### Architecture Overview

The system follows a linear pipeline: **Adapter → NormalizedContent → Domain Chunker → Chunks → Embedder → Store**. Phase 3 adds components at two layers:

1. **Domain layer**: `TasksDomain` and `EventsDomain` chunker implementations
2. **Adapter layer**: Task-source adapters (Apple Reminders, Obsidian tasks/kanban) and event-source adapters (CalDAV/iCalendar, Apple Health)

## Established Patterns

Phase 3 follows patterns established in Phases 1–2. No new architectural patterns are introduced.

### Pattern: Metadata via `extra_metadata`

The adapter→domain metadata contract uses `StructuralHints.extra_metadata` (a `dict[str, object]`). The adapter populates it with domain-specific fields; the domain chunker validates and extracts it. This pattern is established by `EmailAdapter` → `MessagesDomain` (via `MessageMetadata` Pydantic validation) and `ObsidianAdapter` → `NotesDomain` (pass-through propagation).

Phase 3 continues this pattern with two new Pydantic metadata models:

- **`TaskMetadata`** — validated by `TasksDomain` from `extra_metadata`
- **`EventMetadata`** — validated by `EventsDomain` from `extra_metadata`

### Pattern: Domain-specific chunking strategy

Each domain defines its own chunking semantics via `BaseDomain.chunk()`:
- `NotesDomain`: heading-hierarchy splits with code/table atomicity
- `MessagesDomain`: one message per chunk, quoted content stripped

Phase 3 domains follow suit:
- `TasksDomain`: **one task per chunk** (as stated in the stub docstring)
- `EventsDomain`: **one event per chunk** with time-windowed summaries

### Pattern: Optional dependencies with graceful import guards

Adapters with external dependencies use try/except import guards with `HAS_*` flags (see `ObsidianAdapter`, `EmailAdapter`, `RichFilesystemAdapter`). Phase 3 adapters follow this pattern for their respective dependencies.

### Pattern: macOS local client → server REST API

Apple Reminders and Apple Health both follow a **local-client/server-API architecture**. A macOS-native client process exposes Apple framework data over a local REST API, and the Python adapter consumes that API. This mirrors the EmailEngine pattern used by `EmailAdapter`: a headless service abstracts provider-specific access behind a REST interface.

### Pattern: Shared filesystem watching via `_watching.py`

The Obsidian Tasks adapter re-uses the existing `FileSystemWatcher` from `_watching.py`, following the same pattern as `ObsidianAdapter` and `RichFilesystemAdapter`. Each adapter instantiates its own watcher pointed at its own path scope.

## Data Models

### TaskMetadata

```python
class TaskMetadata(BaseModel):
    """Task metadata extracted by task-source adapters."""
    model_config = ConfigDict(frozen=True)

    task_id: str                              # unique identifier from source
    status: str                               # e.g., "open", "completed", "cancelled"
    title: str                                # task title (required)
    due_date: str | None = None               # ISO 8601, optional
    priority: int | None = None               # numeric priority, optional
    dependencies: tuple[str, ...] = ()        # task_ids this depends on
    collaborators: tuple[str, ...] = ()       # people associated with task
    date_first_observed: str                  # ISO 8601, when adapter first saw this task
    source_type: str                          # e.g., "apple_reminders", "obsidian_tasks"
```

Validation: `task_id` and `title` must be non-empty. `due_date` and `date_first_observed` validated as ISO 8601. `status` validated against an allowlist of known values (extensible).

### EventMetadata

```python
class EventMetadata(BaseModel):
    """Event metadata extracted by event-source adapters."""
    model_config = ConfigDict(frozen=True)

    event_id: str                             # unique identifier from source
    title: str                                # event title (required)
    start_date: str | None = None             # ISO 8601, optional
    end_date: str | None = None               # ISO 8601, optional
    duration_minutes: int | None = None       # optional, derived or explicit
    host: str | None = None                   # person facilitating
    invitees: tuple[str, ...] = ()            # other participants
    date_first_observed: str                  # ISO 8601, when adapter first saw this event
```

Validation: `event_id` and `title` must be non-empty. Temporal fields validated as ISO 8601. Model invariant: if both `start_date` and `end_date` are present, `start_date <= end_date`.

Both models are added to `src/context_library/storage/models.py` alongside the existing `MessageMetadata`.

## Component Design

### TasksDomain (`src/context_library/domains/tasks.py`)

**Chunking strategy**: One task per chunk. The stub docstring specifies "one-task-per-chunk strategy with lifecycle state tracking."

Each task adapter yields one `NormalizedContent` per task, with the task description as markdown and `TaskMetadata` fields in `extra_metadata`. The `TasksDomain` chunker:

1. Extracts and validates `TaskMetadata` from `extra_metadata` (same pattern as `MessagesDomain` with `MessageMetadata`)
2. Builds a context header: `"{title} [{status}]"` (optionally with due date: `"{title} [due: {due_date}] [{status}]"`)
3. Emits a single chunk per task (the description/notes body), or splits at sentence boundaries if the description exceeds `hard_limit`
4. Stores `TaskMetadata.model_dump()` as `domain_metadata` on each chunk
5. Returns empty list if the task has no description content

**Status-aware filtering**: The `domain_metadata` includes `status`, enabling retrieval-time filtering via the existing metadata filter path in `retrieve()`. No special storage-layer changes needed.

### EventsDomain (`src/context_library/domains/events.py`)

**Chunking strategy**: One event per chunk. The stub docstring specifies "time-windowed summaries for structured event records."

Each event adapter yields one `NormalizedContent` per event, with the event description/notes as markdown and `EventMetadata` fields in `extra_metadata`. The `EventsDomain` chunker:

1. Extracts and validates `EventMetadata` from `extra_metadata` (same pattern)
2. Builds a context header: `"{title} — {start_date}"` (or `"{title}"` if no start date)
3. Emits a single chunk per event, splitting at sentence boundaries if needed
4. Stores `EventMetadata.model_dump()` as `domain_metadata`
5. Returns empty list if event has no description content

**Time-windowed summaries**: The "time-windowed" aspect from the docstring refers to the context header including temporal context, making vector search results time-aware. The actual time-window grouping (e.g., "meetings this week") is a retrieval-layer concern deferred to Phase 4.

## Adapter Design

### Apple Reminders Adapter (`src/context_library/adapters/apple_reminders.py`)

**Architecture**: macOS client + server adapter API

A macOS-native helper process uses the EventKit framework (via PyObjC or a Swift CLI) to read Reminders data, and exposes it over a local HTTP API. The Python adapter consumes this REST endpoint, mirroring the EmailEngine pattern used by `EmailAdapter`.

**Why a local client**: Apple's Reminders data is only accessible through EventKit on macOS (or CloudKit with iCloud credentials). There is no public REST API. A local intermediary is the only viable path.

**Adapter properties**:
- **Domain**: `Domain.TASKS`
- **adapter_id**: `f"apple_reminders:{account_id}"`
- **PollStrategy**: `PULL` (periodic polling of the local REST API)
- **normalizer_version**: `"1.0.0"`

**Fetch behavior**:
- Calls the local REST API to list reminders (with optional list filter via `source_ref`)
- Supports incremental fetch: passes `source_ref` as ISO 8601 timestamp to filter reminders modified since last fetch
- Converts each reminder to `NormalizedContent`:
  - `markdown`: reminder notes/body text (or title if no notes)
  - `source_id`: `f"{list_name}/{reminder_id}"`
  - `extra_metadata`: `TaskMetadata` fields — `task_id`, `title`, `status` (mapped from Reminder completion state), `due_date`, `priority` (mapped from Reminder priority 1-9), `collaborators` (from shared reminders)

**Dependencies**: `httpx` (HTTP client, already used by `EmailAdapter`)

**Optional dependency group**: `apple-reminders = ["httpx"]` in `pyproject.toml`

### Obsidian Tasks Adapter (`src/context_library/adapters/obsidian_tasks.py`)

**Architecture**: Re-uses `_watching.py` filesystem watcher; parses Obsidian task/kanban file formats directly.

**Scope**: This adapter reads task-formatted content from an Obsidian vault. It handles:
- [Obsidian Tasks plugin format](https://publish.obsidian.md/tasks/): Checkbox items with emoji-based metadata (`📅`, `⏳`, `✅`, `🔺`, etc.)
- [Kanban plugin format](https://github.com/mgmeyers/obsidian-kanban): YAML-frontmatter-tagged markdown files with list items as cards under heading-based lanes

**Adapter properties**:
- **Domain**: `Domain.TASKS`
- **adapter_id**: `f"obsidian_tasks:{vault_path}"`
- **PollStrategy**: `PULL` or `PUSH` (same pattern as `ObsidianAdapter`)
- **normalizer_version**: `"1.0.0"`

**Fetch behavior**:
- Walks vault `.md` files (or receives push events via `FileSystemWatcher`)
- For each file, scans for task-formatted lines (checkbox syntax `- [ ]`, `- [x]`, `- [/]`, etc.)
- Extracts per-task metadata from inline annotations:
  - Due date: `📅 YYYY-MM-DD` or `[due:: YYYY-MM-DD]`
  - Priority: `🔺` (highest), `⏫` (high), `🔼` (medium), `🔽` (low) or `[priority:: ...]`
  - Dependencies: `⛔ task-id` or `[depends:: ...]`
  - Status: mapped from checkbox marker (`[ ]` = open, `[x]` = completed, `[/]` = in-progress, `[-]` = cancelled)
- For kanban files (detected via frontmatter `kanban-plugin: basic`):
  - Heading = lane/status, list items under heading = task cards
  - Status derived from lane heading (e.g., "Done" → completed)
- Each task yields one `NormalizedContent` with task body as markdown and `TaskMetadata` in `extra_metadata`

**Dependencies**: `python-frontmatter` (already required by `obsidian` group), `watchdog` (already in `obsidian` group)

**Optional dependency group**: Re-uses `obsidian` group — no new dependencies needed.

### CalDAV/iCalendar Adapter (`src/context_library/adapters/caldav.py`)

**Architecture**: Standards-based calendar adapter using CalDAV (RFC 4791) and iCalendar (RFC 5545).

CalDAV is the dominant open protocol for calendar access, supported by Google Calendar, Apple Calendar, Fastmail, Nextcloud, and most self-hosted calendar servers. Using CalDAV directly (rather than provider-specific APIs) provides the broadest compatibility.

**Adapter properties**:
- **Domain**: `Domain.EVENTS`
- **adapter_id**: `f"caldav:{calendar_url}"`
- **PollStrategy**: `PULL` (periodic polling; CalDAV supports `sync-token` for efficient delta sync)
- **normalizer_version**: `"1.0.0"`

**Fetch behavior**:
- Connects to CalDAV server with credentials (URL + username + password/token)
- Lists calendars, optionally filtered by `source_ref` (calendar name or path)
- Fetches events via `REPORT` request with date-range filter (or `sync-token` for incremental)
- Parses iCalendar (`.ics`) data for each event
- Converts each event to `NormalizedContent`:
  - `markdown`: event description/notes (DESCRIPTION property) or summary if no description
  - `source_id`: `f"{calendar_name}/{event_uid}"`
  - `extra_metadata`: `EventMetadata` fields — `event_id` (UID), `title` (SUMMARY), `start_date` (DTSTART), `end_date` (DTEND), `duration_minutes` (computed from DURATION or DTSTART/DTEND), `host` (ORGANIZER), `invitees` (ATTENDEE list)

**Dependencies**: `caldav` (Python CalDAV client library), `icalendar` (iCalendar parser)

**Optional dependency group**: `caldav = ["caldav", "icalendar"]` in `pyproject.toml`

A macOS-native helper process uses the HealthKit framework to read health event data (workouts, activity summaries) and exposes it over a local HTTP API. The Python adapter consumes this REST endpoint.

**Why a local client**: HealthKit data is only accessible through Apple's HealthKit framework on macOS/iOS. There is no public REST API. Export-based approaches (XML export from Health app) are manual and batch-only.

## Scalability Design

### Incremental Ingestion

All four adapters support incremental fetch via the existing `source_ref` timestamp mechanism:
- **Apple Reminders / Apple Health**: Pass ISO 8601 timestamp to local API for "modified since" filtering
- **Obsidian Tasks**: Change detection handled by the differ (hash-based); push mode via `FileSystemWatcher` provides real-time updates
- **CalDAV**: Native `sync-token` support for efficient delta sync without re-fetching all events

### One-Entity-Per-Chunk Scaling

The one-task-per-chunk and one-event-per-chunk strategies produce small, focused chunks. This means:
- Embedding cost scales linearly with entity count (not content volume)
- LanceDB vector search remains efficient since chunks are uniform in size
- The differ's set-based hash comparison efficiently handles entity additions/removals

### No New Storage Requirements

Both domains use the existing `Domain.EVENTS` and `Domain.TASKS` enum values already defined in the schema. The SQLite `adapters.domain` CHECK constraint already includes these values. No schema migration is needed.

## Security Considerations

### Local REST API Clients

The Apple Reminders and Apple Health adapters communicate with local macOS helper processes over HTTP. Security measures:
- **Bind to localhost only** (`127.0.0.1`): The helper process must not expose its API on network interfaces
- **No credential storage in the adapter**: OAuth/API tokens managed by the macOS helper process via Keychain
- The adapter configuration stores only the API endpoint URL and an optional API key for the local service

### CalDAV Credentials

CalDAV requires authentication (username + password or OAuth token). The adapter accepts credentials at construction time but does not persist them. Credential management (secure storage, rotation) is the caller's responsibility, consistent with the `EmailAdapter` pattern.

### Input Validation

All adapter output passes through Pydantic validation:
- `TaskMetadata` and `EventMetadata` validate field types, non-empty constraints, and ISO 8601 formats
- The existing `StructuralHints.extra_metadata` → Pydantic model validation pattern (from `MessagesDomain`) prevents malformed metadata from reaching the storage layer
