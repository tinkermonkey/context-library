## System Architecture

### Architecture Style: Layered + Hexagonal Hybrid

The existing Architecture.md defines a clean layered system with six concerns: **Adapters**, **Domains**, **Core**, **Storage**, **Retrieval**, and **Scheduler**. This maps naturally to a `src` layout using Python's namespace packaging convention (`src/context_library/`).

The architecture is a hybrid of:
- **Layered architecture** — data flows unidirectionally through fetch → normalize → diff → chunk → embed → store
- **Ports and adapters (hexagonal)** — the `BaseAdapter` and `BaseDomain` abstract classes define inward-facing ports; concrete adapters are outward-facing implementations

This is appropriate for the problem domain. A personal RAG system with heterogeneous inputs needs the adapter abstraction to isolate source-specific complexity, and the layered pipeline provides a clear mental model for debugging and extension.

### Package Layout

```
src/context_library/
├── __init__.py
├── core/                   # Pipeline orchestration, diffing, embedding, versioning, chunking
├── domains/                # Semantic layer — chunking strategies per content type
├── adapters/               # Source-specific fetch/normalize implementations
├── storage/                # SQLite document store + LanceDB vector store
├── retrieval/              # Query interface, reranking, provenance
└── scheduler/              # Polling and push-based ingestion triggers
```

The `src/` layout (as opposed to flat `context_library/`) is the correct choice here. It prevents accidental imports of the development package during testing and is the standard recommended by the Python Packaging Authority.

### Dependency Direction

Dependencies flow inward:

```
scheduler → core (pipeline)
adapters → core (types: RawContent, NormalizedContent)
core → domains (chunking delegation)
core → storage (read/write)
retrieval → storage (query)
```

No layer depends on `adapters` or `scheduler`. The `Pipeline` class in `core/` holds references to abstract interfaces, not concrete implementations. This is already encoded in the Architecture.md class diagram and must be preserved in the scaffold.

## Scalability Design

### Storage Scalability

The dual-store architecture (SQLite + LanceDB) is well-suited for the personal-scale use case described in the design documents. Key scaling properties:

| Dimension | SQLite | LanceDB |
|-----------|--------|---------|
| Write throughput | WAL mode enables concurrent reads during writes; sufficient for adapter polling rates | Bulk insert via Lance columnar format |
| Read throughput | Indexed queries on `chunk_hash`, `source_id`, `domain` | IVF-PQ indexing after ~10K chunks; brute-force below that |
| Storage ceiling | Practical limit ~1TB; well beyond personal use | Disk-backed columnar format; grows with chunk count |
| Recovery | Standard SQLite backup/WAL | Fully rebuildable from SQLite |

The persistence design explicitly states that LanceDB is **derived and disposable** — this is a strong scalability property because it means the embedding model can be changed without data loss and the vector index can be rebuilt offline.

### Extension Points

- **New adapters**: Implement `BaseAdapter`, declare a domain, register. No core changes.
- **New domains**: Subclass `BaseDomain`, define chunking strategy and metadata schema. Requires adding a `DomainType` enum value.
- **Embedding model swap**: Re-embed from SQLite. The `Embedder` class is explicitly designed as a swappable interface.

### What the Scaffold Must Not Introduce

The scaffold should not introduce any runtime abstractions for scaling (connection pools, worker queues, caching layers). These are premature for a single-user local application. The architecture documents make no mention of multi-tenancy or distributed operation, and the scaffold should not design for these.

## Established Patterns

### Patterns Identified in the Design Documents

1. **Content-addressed storage** — Chunks are identified by hash of their normalized content. This is the core versioning mechanism and the join key between SQLite and LanceDB.

2. **Source of truth + derived index** — SQLite is authoritative; LanceDB is a projection. This pattern simplifies consistency reasoning and failure recovery.

3. **Abstract base + concrete implementations** — `BaseAdapter` and `BaseDomain` define contracts. Concrete classes implement source-specific or domain-specific behavior.

4. **Schema versioning via PRAGMA user_version** — The persistence design specifies `schema_version` tracking for SQLite migrations. Using `PRAGMA user_version` is the lightest-weight approach consistent with the design's preference for simplicity.

5. **JSON-in-column for domain metadata** — Domain-specific fields are stored as JSON in `chunks.domain_metadata` rather than in separate tables. This avoids premature schema normalization while preserving queryability.

### Patterns the Scaffold Must Follow

- **`pyproject.toml` as sole configuration** — No `setup.py`, `setup.cfg`, or `requirements.txt`. Modern Python packaging via PEP 621.
- **Optional dependency groups for adapters** — Each adapter ecosystem (Gmail, Spotify, Todoist) has its own extra. Core dependencies are mandatory; adapter dependencies are opt-in.
- **`src` layout** — Package under `src/context_library/`, tests at project root under `tests/`.

### Patterns to Avoid

- **No dependency injection framework** — The Pipeline class directly composes its dependencies. A DI framework would be over-engineering for this scale.
- **No ORM** — The persistence design uses raw SQL. SQLite access should use `sqlite3` from the standard library, not SQLAlchemy.
- **No async** — Nothing in the architecture documents suggests async operation. The scheduler polls; adapters fetch synchronously. Introducing `asyncio` in the scaffold would add complexity with no stated benefit.
