**Description**:
Based on consolidated PR review, address the following in **Poller**:

- **Catch-all with no retry, failure tracking, or escalation**: `/workspace/src/context_library/scheduler/poller.py:108-114` — All pipeline errors are caught and logged with no retry or circuit breaker. `update_last_fetched_at` is inside the same try block after `ingest`, so if ingest succeeds but the update fails, the source enters an infinite re-ingestion loop.
- **`_run()` does not pass `source_ref` to `adapter.fetch()`**: `/workspace/src/context_library/scheduler/poller.py` — FR-1.2 requires the poller to invoke `adapter.fetch(source_ref)` for each source. The implementation delegates to `self._pipeline.ingest(adapter, chunker)` without forwarding `source_ref`. Incremental ingestion (e.g., EmailAdapter's `since` parameter) does not work through the poller.
- **First tick delayed by full `tick_interval`**: `/workspace/src/context_library/scheduler/poller.py:87-90` — `_run()` uses `Event.wait(timeout=tick_interval)` before the first `_tick()`. With default 60s interval, a newly started poller waits 60 seconds before its first poll.
- **`stop()` joins thread without timeout**: `/workspace/src/context_library/scheduler/poller.py:82-85` — If `_tick()` hangs on a network call, `stop()` blocks indefinitely.
